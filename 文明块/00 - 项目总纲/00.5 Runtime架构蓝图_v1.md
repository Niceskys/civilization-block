## 1. 核心原则（不可违反）

### 1.1 模拟层唯一权威（Simulation Layer Single Authority）

所有游戏逻辑状态必须存在于 Simulation Layer：

- 所有资源数量与类型
- 所有建筑实例状态
- 所有NPC实例状态
- 所有任务进度
- 所有时间状态
- 所有随机数生成
- 所有配方进度
- 所有怪物实例状态

UI 层只能读取 Simulation Layer 输出的 ViewState，不可直接修改任何游戏状态。

### 1.2 命令系统唯一入口（Command Bus Single Entry）

禁止任何系统直接修改 Simulation Layer 状态。所有修改必须通过 Command：

```
Command {
  command_id: string,          // 唯一命令ID（UUID）
  player_id: string,           // 玩家ID（单机固定值为 "local_player"）
  type: string,                // 命令类型（带namespace）
  payload: object,             // 命令负载
  timestamp: number,           // 客户端生成时间戳
  sequence: number             // 客户端序列号（用于排序和去重）
}
```

所有命令必须经过 `Simulation.ValidateCommand(command)` → `Simulation.ExecuteCommand(command)` 原子流程。

### 1.3 事件系统唯一输出（Event Stream Single Output）

Simulation Layer 只通过 Event Stream 向外输出状态变更通知：

```
Event {
  event_id: string,            // 唯一事件ID
  event_type: string,          // 事件类型（带namespace）
  source: string,              // 事件来源（command_id 或 系统自动触发）
  timestamp: number,           // 模拟层时间戳
  payload: object,             // 事件负载
  version: number              // 数据结构版本号
}
```

❌ UI 之间不可直接互调。❌ 系统之间不可直接互调。

### 1.4 三层分离（Three-Layer Separation）

所有对象必须拆分为：

| 层 | 说明 | 存储位置 |
|---|------|---------|
| Definition（静态定义） | 类型、配方、数值模板、解锁条件 | 配置文件 / mod 文件 |
| State（运行状态） | 实例数据、当前HP、当前位置、当前等级 | 存档 / 内存 |
| View（表现状态） | UI锚点、动画状态、选中等临时状态 | UI层本地 |

### 1.5 ID系统（ID System）

所有持久对象使用带 namespace 的字符串ID：

```
格式: {namespace}:{type}:{instance_id}

示例:
  building:core:farm_001
  npc:core:zhangshan_042
  item:core:iron_ingot
  command:core:cmd_build_20260617_001
  quest:core:quest_water_01
  mod:example_mod:windmill_def
```

禁止：数字ID、场景路径、内存引用、无前缀短名。

### 1.6 确定性要求（Determinism）

Simulation Layer 必须是确定性的（deterministic）：

- 相同的初始状态 + 相同的命令序列 → 完全相同的结果状态
- 随机数必须通过种子化RNG（Simulation.SeededRNG）
- 禁止依赖：系统时钟（使用游戏内时间）、网络延迟、浮点数精度差异、未定义循环顺序的哈希表遍历

### 2.3 Tick Pipeline（时间推进管道）

FACT：当前时间系统定义在 `1.3 时间系统规则.md`，以"白天10分钟 + 夜晚10分钟"为周期。生产、NPC行为、怪物生成都在固定时间边界触发，但没有定义统一的 Tick 推进架构。

PLAN：定义 Tick 推进管道：

```
每个游戏 Tick（1游戏分钟）按以下顺序执行：

1. TimeSystem.Tick()                    // 推进游戏时间
2. CommandBus.FlushPending()            // 处理等待中的命令队列
3. ResourceSystem.Tick()                // 资源生产/消耗（每分钟批次）
4. BuildingSystem.Tick()                // 建筑状态更新（建造进度、耐久、自动运转）
5. NPCAISystem.Tick()                   // NPC行为（工作寻路、休息、社交）
6. QuestSystem.Tick()                   // 任务进度检查
7. CombatSystem.Tick()                  // 怪物AI、攻击结算
8. EventSystem.CheckTriggers()          // 检查事件触发条件
9. EventStream.Flush()                  // 输出本Tick产生的事件
```

ASSUMPTION：每分钟 Tick 是合理的默认粒度。某些子系统（如战斗）可能需要子Tick精度，但当前阶段设计为在 Tick 内通过逻辑分帧解决。

### 2.4 SeededRNG（种子化随机数）

PLAN：

```
SeededRNG {
  seed: number,              // 当前种子
  call_count: number,        // 已调用次数（保证确定性）
  
  Next(): number             // 返回 [0, 1) 的伪随机数
  NextRange(min, max): number // 返回 [min, max] 的整数
  NextChoice(array): any     // 从数组中随机选一个
  Fork(): SeededRNG          // 生成子RNG（用于并发事件）
}
```

RNG 种子初始化：
- 世界种子 + "world" → WorldRNG
- 世界种子 + "npc" → NPCRNG
- 世界种子 + "events" → EventRNG
- 世界种子 + "combat" → CombatRNG

ASSETION：所有随机事件必须通过 SeededRNG，禁止使用语言内置的 Math.random()。

## 3. 命令总线（Command Bus）

### 3.1 架构

```
Client Layer (UI)
     │
     ▼ 发送 Command
CommandBus
     │
     ├─ Validate (模拟层验证)
     │   ├─ 命令格式验证
     │   ├─ 权限验证（联机时）
     │   ├─ 前置条件验证（资源是否足够、建筑是否存在等）
     │   └─ 防重复验证（command_id去重）
     │
     ├─ Enqueue (加入待处理队列)
     │
     └─ Execute (在下一个 Tick 内执行)
          │
          └─ 分发到对应 Sub-System Handler
               └─ 返回 Events
```

### 3.2 命令类型定义

PLAN：所有命令按命名空间和类型分类。

```

// --- 资源命令 (namespace: resource) ---

// --- 任务命令 (namespace: quest) ---
```

### 3.3 命令验证（ValidateCommand）

PLAN：每个命令类型对应一个验证器函数：

```
ValidateCommand(command): ValidationResult {
  1. 格式校验：字段类型、必填字段、命名空间格式
  2. 幂等校验：command_id 是否已执行过（防重放）
  3. 存在性校验：引用的实例ID是否存在
  4. 前置条件校验：资源是否足够、建筑等级是否达标、冷却时间是否已过
  5. 权限校验：联机模式下检查 player_id 是否有权限
  6. 模组校验：调用所有注册的 mod_command_validators
}
```

### 3.4 命令执行（ExecuteCommand）

PLAN：每个命令类型对应一个处理器函数：

```
ExecuteCommand(command): Event[] {
  1. 锁定相关资源（防止并发冲突）
  2. 修改 GameState
  3. 产生 0~N 个 Event
  4. 释放锁定
  5. 返回 Event 列表
}
```

### 3.5 命令日志

PLAN：所有已执行的命令必须在存档中记录，用于：

- 回放调试（Replay Debugging）
- 联机同步校验
- 反作弊审计
- 数据恢复

## 4. 事件流（Event Stream）

### 4.1 架构

```
Simulation Layer
     │
     └─ EmitEvent(event) ──→ EventStream
                                │
                                ├─ 写入 EventLog（持久化）
```

### 4.3 事件订阅机制

PLAN：

```
EventStream.Subscribe(event_type_filter, callback, priority)
  - event_type_filter: 事件类型前缀匹配（如 "building." 匹配所有建筑事件）
  - callback: (event) => void
  - priority: number（默认0，越大越先执行）

用途：
  1. UI层订阅 → 刷新UI视图
  2. 模组订阅 → 触发模组逻辑
  3. 存档系统订阅 → 标记脏状态（需要存档）
  4. 联机同步订阅 → 转发到其他客户端
```

### 4.4 事件驱动的自动系统

部分系统可以注册"事件→命令"自动转换：

```
示例：
  事件: resource.changed (食物 < 10)
  → 自动生成命令: quest.activate { quest_id: "food_emergency" }
  
  事件: monster.spawned (definition_id = "boss")
  → 自动生成命令: quest.activate { quest_id: "boss_fight" }
```

这些自动转换必须经过明确注册，且可在模组配置中覆盖。

## 5. 存档系统（Save System）

### 5.1 设计原则

1. **存档 = GameState 序列化**：存档是完整的权威状态快照
2. **版本向前兼容**：旧存档可在新版本加载（自动迁移）
3. **冗余校验**：CRC + 备份存档
4. **模组感知**：存档标记所依赖的模组列表
5. **回放能力**：命令日志可作为"回放文件"用于调试

### 5.2 存档结构

PLAN：

```
SaveFile {
  format_version: string,          // 存档格式版本号（如 "1.0"）
  game_version: string,            // 游戏版本号
  timestamp: number,               // 存档时间（UTC时间戳）
  
  // --- 模组依赖 ---
  required_mods: ModDependency[],  // 此存档依赖的模组列表
  
  // --- 完整状态 ---
  game_state: GameState,           // 序列化的完整GameState
  
  // --- 命令日志（用于回放） ---
  command_log: ExecutedCommand[],  // 从存档开始到存档时刻的所有命令
  
  // --- 校验 ---
  checksum: string,                // GameState 的 CRC 校验
  signature: string | null,        // 数字签名（联机防篡改）
}

ModDependency {
  mod_id: string,                  // 带namespace
  version: string,                 // 依赖的模组版本
  required: boolean                // true=必须加载, false=可选
}
```

### 5.3 存档操作接口

PLAN：

```
SaveSystem {
  Save(slot_name: string, metadata: SaveMetadata): Result
  
  Load(slot_name: string): Result<SaveFile>
  
  // 自动存档（每隔N游戏日或现实时间）
  AutoSave(): Result
  
  // 存档迁移（旧版本 → 新版本）
  MigrateSave(old_save: SaveFile, target_version: string): Result<SaveFile>
  
  // 校验存档完整性
  ValidateSave(save: SaveFile): ValidationResult
  
  // 列出所有存档及其元数据
  ListSaves(): SaveMetadata[]
  
  // 删除存档
  DeleteSave(slot_name: string): Result
}

SaveMetadata {
  slot_name: string,
  save_time: number,
  game_time: GameTimeState,
  player_summary: {
    day: number,
    population: number,
    building_count: number,
    resource_summary: Record<string, number>
  },
  required_mods: string[],
  screenshot_uri: string | null
}
```

### 5.4 存档迁移

FACT：当前游戏规则处于早期版本，多个数值表（6.1, 6.2, 6.3）状态为"补充中"。随着游戏版本迭代，存档格式必然需要迁移。

PLAN：

```
迁移机制：
1. 每个存档记录 format_version
2. 游戏启动时检查存档版本号
3. 若版本 < 当前版本，按顺序应用迁移脚本
4. 迁移脚本必须是确定性的
5. 迁移后重新计算 checksum

迁移脚本示例：
  v1.0 → v1.1: 新增 ResourceState.production_rates 字段
  v1.1 → v1.2: NPCState 新增 mood 字段，默认值为 {current: 0, factors: []}
  v1.2 → v2.0: 建筑状态拆分出 ProductionState 子结构
```

### 5.5 脏状态标记系统

PLAN：为避免每 Tick 全量存档，实现脏状态标记：

```
DirtyTracker {
  MarkDirty(path: string): void       // 标记某个状态路径已修改
  IsDirty(path: string): boolean      // 检查是否被修改
  SnapshotDirty(): string[]           // 获取所有脏路径并清空
  ClearAll(): void                    // 清空所有标记
}

示例：
  building.upgrade 命令执行后
  → DirtyTracker.MarkDirty("buildings.instance_001")
  → AutoSave 时只序列化脏数据，全量写入
```

## 6. 模组注册表（Mod Registry）

### 6.1 设计原则

1. **只做规范，不做实现**：当前阶段只设计模组能力接口，不实现模组运行环境
2. **沙箱隔离**：模组在定义好的接口范围内操作，不能访问未授权的系统
3. **依赖声明**：模组明确声明依赖关系
4. **版本兼容性**：通过版本号声明兼容的游戏版本

### 6.2 Mod Manifest 结构

PLAN：

```
mod:example_mod/

├── hooks/                 # 钩子脚本
│   ├── command_hooks.js   # 命令钩子
│   └── event_hooks.js     # 事件钩子
└── assets/                # 资源文件
    ├── models/
    ├── textures/
    └── localizations/
```

```

  // 检查模组新增的前置条件
```

### 6.6 模组加载流程

PLAN：

```

  1. 验证 manifest.json 完整性

  3. 从 DefinitionRegistry 移除定义
  4. 从已加载模组列表移除
}
```

## 7. 网络抽象层（Network Abstraction Layer）

### 7.1 设计原则

1. **单机 = 本地模拟**：单机模式下，Client 和 Simulation Layer 在同一进程，通过本地通信
2. **联机 = 远程模拟**：联机模式下，Simulation Layer 运行在服务器，Client 通过 NAL 发送命令、接收事件
3. **透明切换**：架构设计支持单机/联机无代码修改切换

### 7.2 通信协议

PLAN：

```
// --- 客户端 → 服务器 ---
// 客户端通过 WebSocket / UDP 发送序列化命令
Client → Server: CommandMessage {
  type: "command",
  payload: Serialized<Command>,
  client_tick: number,
  checksum: string
}

// 服务器 → 客户端 ---
// 服务器广播状态变更
Server → Client: EventMessage {
  type: "event",
  payload: Serialized<Event>[],
  server_tick: number,
  authoritative_state_hash: string
}

Server → Client: StateSnapshot {
  type: "snapshot",
  payload: Serialized<Partial<GameState>>,
  server_tick: number
}

// --- 连接建立 ---
Client → Server: ConnectMessage {
  type: "connect",
  player_id: string,
  auth_token: string,
  client_version: string,
  mod_list: string[]
}

Server → Client: ConnectAck {
  type: "connect_ack",
  player_id: string,
  server_version: string,
  accepted_mods: string[],
  initial_state: Serialized<GameState>,
  server_tick: number
}
```

### 7.3 网络抽象接口

PLAN：

```
// NAL 接口（单机和联机两种实现）
interface NetworkAbstractionLayer {
  // 发送命令（通用接口）
  SendCommand(command: Command): void
  
  // 注册事件处理器
  OnEvent(handler: (event: Event) => void): void
  
  // 获取当前连接状态
  GetConnectionState(): ConnectionState
  
  // 延迟统计
  GetLatency(): number
  
  // 连接管理
  Connect(server_url: string, credentials: AuthInfo): Promise<Result>
  Disconnect(): void
  IsConnected(): boolean
}

// --- 单机实现 ---
class LocalNAL implements NetworkAbstractionLayer {
  // 直接调用本地 Simulation Kernel
  // 没有网络传输
  // 延迟 = 0
}

// --- 联机实现 ---
class RemoteNAL implements NetworkAbstractionLayer {
  // 通过 WebSocket 连接远程服务器
  // 序列化/反序列化命令和事件
  // 处理网络波动和重连
  // 客户端预测（可选）
}
```

### 7.4 客户端预测与服务器权威

PLAN：在联机场景下定义客户端预测策略。

```
// P0（首发必需）：权威先读
// UI 发送命令后，等待服务器确认再更新
// 延迟敏感度低的操作（建造、升级、研究）

// P1（体验优化）：乐观更新 + 服务器纠正
// UI 发送命令后，立即在本地模拟执行结果
// 服务器命令处理结果返回后，纠正任何偏差
// 延迟敏感度高的操作（NPC移动、战斗）

// P2（后续）：状态同步 + 差值更新
// 服务器定期广播全量或差值状态
// 客户端插值平滑显示
// 适用于低频率变动的状态
```

### 7.5 防作弊架构（联机模式）

PLAN：

```
1. 服务器权威模拟：
   - 所有 Command 发送到服务器执行
   - 服务器 Simulation 是唯一合法状态
   - 客户端不运行权威模拟

2. 命令审计：
   - 每个命令附带 client_tick 和 checksum
   - 服务器验证命令合法性（资源校验、建筑存在性校验）
   - 记录的命令日志用于回放审核

3. 存档校验：
   - 自动存档附带 CRC 和服务器签名
   - 修改存档 → 校验失败
```

## 9. 迁移路径与风险分析

### 9.1 迁移优先级

| 阶段 | 组件 | 说明 | 建议优先级 |
|------|------|------|-----------|
| P0 | GameState 状态树定义 | 定义所有状态结构，为其他系统提供基础 | 必须先做 |
| P0 | CommandBus | 所有状态修改的入口 | 必须先做 |
| P0 | Command 类型定义 | 定义所有允许的命令 | 必须先做 |
| P1 | EventStream | 状态变更通知系统 | P0完成后 |
| P1 | Event 类型定义 | 定义所有事件 | P0完成后 |
| P1 | Definition 与 State 分离 | 将现有文件拆分为静态定义和运行时状态 | P0完成后 |
| P1 | ID系统迁移 | 所有对象使用 namespace 格式ID | P0完成后 |
| P2 | Tick Pipeline | 统一的时间推进管道 | 子系统就绪后 |
| P2 | SeededRNG | 种子化随机数 | P0完成后 |
| P2 | SaveSystem | 版本化存档系统 | 状态树稳定后 |
| P3 | ModRegistry | 模组注册表 | 核心系统稳定后 |
| P3 | NetworkAbstractionLayer | 网络抽象层 | 长期目标 |

### 9.2 风险分析

| 风险 | 等级 | 说明 | 缓解措施 |
|------|------|------|---------|
| 现有文件完整重写成本 | **高** | 现有约40+个A级源文件需要映射到新的三层架构 | 分阶段迁移，P0只定义接口，不重写内容 |
| UI与Simulation耦合 | **高** | 当前UI设计直接描述状态修改逻辑，未分离 | 先定义CommandBus，将UI中的"修改逻辑"改为"发送命令" |
| 确定性与现有随机事件冲突 | **中** | 现有事件系统使用"运行时随机"，而非种子化RNG | 新系统从零实现SeededRNG，旧逻辑逐步迁移 |
| 模组系统过度设计 | **中** | 在游戏尚未实现核心系统时定义模组接口，可能导致后续修改 | ModRegistry只定义schema和接口，不做实现 |
| 网络层引入延迟 | **低** | 单机模式零延迟，联机模式才需要处理 | NAL抽象接口，单机实现不做任何网络处理 |
| 存档版本化复杂度 | **中** | 迁移脚本需要维护，版本号需要管理 | 所有状态结构附带version字段，迁移脚本以函数数组管理 |
| 命令系统与现有规则冲突 | **中** | 现有文件中的"建筑可直接写逻辑"与命令系统冲突 | 在00.1规则中补充声明：所有实现必须通过CommandBus |
| 权威状态树大小 | **低** | 每Tick保存完整状态可能性能问题 | 使用DirtyTracker只序列化变更部分 |

| Command | 对游戏状态的修改请求，必须是经过验证的 |
| Event | 状态变更后的通知，不可修改状态 |
| Tick | Simulation Layer 的最小时间推进单位（1游戏分钟） |
| Definition | 静态数据定义（不变的数据模板） |
| State | 运行时实例数据（可变的存档数据） |
| View | UI层表现数据（不存档） |
| NAL | NetworkAbstractionLayer，网络抽象层 |
| SeedRNG | 基于种子的确定性随机数生成器 |
| DirtyTracker | 脏状态标记系统，用于优化存档性能 |
| ModRegistry | 模组注册表，管理模组加载和钩子 |

### DESIGN NOTES（设计备注）

1. **为什么 GameState 不将 UI 状态包含在内？**
   UI状态（如面板打开/关闭、选中状态、悬停高亮）不改变游戏逻辑结果，不应进入权威状态树。存档也不应包含UI状态。

2. **为什么 Tick 粒度是 1 游戏分钟而非实时？**
   1游戏分钟 = 现实30秒（1倍速）。此粒度足够覆盖资源生产、NPC行为、建筑维护等低频更新。战斗系统可通过子Tick或逻辑帧解决高频需求。

3. **namespace 格式为何不从 "core:" 简化为空？**
   即使单机版本不使用namespace，引入模组系统后 namespace 立刻成为必需品。初始使用 namespace 可避免后续大规模ID重构。

4. **Command 与 Event 为何使用不同结构而非同一结构？**
   方向不同：Command 是"请求修改"，Event 是"通知结果"。混淆二者会导致系统耦合。Command 需要验证，Event 不需要验证。Command 可以被拒绝，Event 不会被拒绝。

5. **自动事件→命令转换（4.4节）是否违反"命令是唯一入口"原则？**
   不违反。自动转换本质是系统内部触发新的Command，这些Command仍然经过 ValidateCommand → ExecuteCommand 管道。区别在于触发源从"用户输入"变为"事件匹配器"。

6. **存档中的 command_log 是否必要？**
   是。Command log 提供回放能力、联机状态校验、冲突恢复和反作弊审计。存档大小增加（约每秒几KB）对本地存档是可接受的。
