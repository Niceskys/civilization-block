# 00.7 核心框架质量与分工蓝图 v1
> 建立日期：2026-06-17
> 用途：定义核心框架完成时机、质量目标、单机/联机/模组边界、调试闭环和多模型分工规则。
> 本文件属于项目总纲层实现蓝图；不直接覆盖任何A级玩法规则。

---

## 一、核心目标

《文明块》的工程目标不是单纯减少 bug，而是让 bug 具有明确边界、明确证据和明确入口。

最终完工时允许存在少量缺陷，但必须满足：

1. 能定位到是哪条 Command、哪个 Tick、哪个系统或哪个定义数据引发问题；
2. 能通过 CommandLog、EventLog、StateHash 或存档快照复现问题；
3. 核心状态不能被 UI、模组脚本或客户端网络消息绕过校验直接修改；
4. 单机和联机使用同一套 Simulation 规则；
5. DeepSeek 等辅助模型只能在固定 schema 和低风险文件中工作，不能改核心架构。

---

## 二、初始框架完成时机

初始框架必须在"前30分钟核心玩法原型"之前完成到可执行状态。

| 阶段 | 完成目标 | 不允许拖后 |
|------|----------|------------|
| 框架 v1 | 本地单机可通过 GameSession 驱动 Simulation；Command/Event/Save/Definition 可闭环 | 大量填建筑、NPC、资源前 |
| 框架 v1.5 | 模组 manifest、定义校验、状态诊断、回放测试基本完成 | 首个可玩版本前 |
| 框架 v2 | RemoteGameSession、服务器权威、模组列表校验、快照同步、断线重连策略完成 | 正式联机开发前 |

若玩法内容已经大量进入，但框架仍未完成，后续 bug 会从"局部错误"升级为"系统性返工"。

## 三、核心架构边界

### 3.1 唯一状态权威

`GameState` 是运行时唯一权威状态树。

禁止：

- UI 直接修改 `GameState`；
- 子系统绕过 `CommandBus` 修改其他系统状态；
- 客户端联机时直接提交状态差异；
- 模组脚本直接写入运行时状态。

允许：

- `Simulation.ExecuteCommand` 在校验通过后修改状态；
- `ISimulationSystem.Tick` 在固定 Tick 顺序中修改所属状态；
- `SaveMigration` 在读档时修复缺失字段和旧结构；
- 测试工具在隔离测试环境中构造初始状态。

### 3.2 唯一修改入口

所有玩家操作、系统自动操作和模组扩展操作都必须转成 `CommandEnvelope`。

基本链路：

```text
UI / Client / Mod
  -> CommandEnvelope
  -> CommandBus.Validate
  -> ICommandHandler.Execute
  -> GameEvent[]
  -> EventStream
  -> View / Log / Network
```

### 3.3 唯一输出链路

所有状态变化都必须产生可观察事件。

事件至少用于：

- UI 刷新；
- 调试面板；
- 联机广播；
- 回放测试；
- 存档审计；
- bug 复现。

## 四、单机与联机模式

### 4.1 模式选择

玩家入口分为：

| 模式 | Runtime 形态 | 说明 |
|------|--------------|------|
| 单机 | `LocalGameSession` | 本地直接持有并推进 Simulation |
| 创建服务器 | `ServerGameSession` | 本机或远端启动权威 Simulation |
| 加入服务器 | `RemoteGameSession` | 客户端只发送 Command 并接收 Event/Snapshot |

UI 只依赖统一 `IGameSession`，不得直接依赖 `Simulation`。

### 4.2 服务器权威原则

联机时：

1. 服务器持有唯一可信 `GameState`；
2. 客户端只发送 `CommandEnvelope`；
3. 服务器负责校验资源、建筑、NPC、权限、模组版本和命令序列；
4. 服务器广播 `GameEvent` 和必要的 `StateSnapshot`；
5. 客户端发现 `StateHash` 不一致时，必须请求快照修复。

### 4.3 单机不能成为特殊分支

单机只是网络层为本地直连：

```text
LocalGameSession.SendCommand(command)
  -> Simulation.ExecuteCommand(command)
```

不能为单机写一套绕过 CommandBus 的快捷逻辑。

## 五、模组系统边界

### 5.1 模组分阶段

| 阶段 | 允许能力 | 禁止能力 |
|------|----------|----------|
| M0 数据模组 | 新增建筑、资源、配方、任务、事件定义 | 脚本、Tick、直接改状态 |
| M1 命令扩展 | 注册新 Command 类型和校验器 | 覆盖核心命令默认规则 |
| M2 事件钩子 | 监听事件并提交新 Command | 直接写 GameState |
| M3 Tick 钩子 | 受限 Tick 扩展 | 非确定性随机、系统时间、网络 IO |

首个可玩版本只建议实现 M0。

### 5.2 模组 manifest 必填字段

```json
{
  "mod_id": "example_mod",
  "name": "Example Mod",
  "version": "1.0.0",
  "min_game_version": "0.1.0",
  "dependencies": {},
  "content_types": ["buildings"],
  "checksum": "sha256..."
}
```

联机进入服务器前必须校验：

- `mod_id` 一致；
- `version` 一致；
- `checksum` 一致；
- 加载顺序一致；
- 依赖关系一致。

### 5.3 模组 ID 规则

模组新增内容必须使用 namespace：

```text
building:example_mod:windmill
item:example_mod:flour
recipe:example_mod:grind_wheat
command:example_mod:set_windmill_mode
event:example_mod:windmill_overloaded
```

禁止模组写入 `core` namespace。

## 六、防 bug 设施

### 6.1 必备调试链路

每次操作都应可展开为：

```text
Command
  -> EnvelopeValidation
  -> HandlerValidation
  -> Execute
  -> Events
  -> StateHash
  -> InvariantCheck
```

### 6.2 必备工具

| 工具 | 用途 |
|------|------|
| `CommandLog` | 记录所有已接受命令，用于回放 |
| `EventLog` | 记录所有状态变化结果 |
| `StateHash` | 对比单机回放或联机同步是否一致 |
| `InvariantCheck` | 检查资源负数、建筑重叠、悬空引用等核心不变量 |
| `DefinitionValidator` | 启动时检查配置和模组定义是否合法 |
| `ReplayTest` | 同一初始状态 + 同一命令序列必须得到同一状态 |

### 6.3 核心不变量

任何 Tick 后至少检查：

1. 资源数量不能小于 0；
2. 建筑实例和施工任务不能占用同一地皮同一层；
3. 施工任务引用的建筑 ID 不能重复；
4. 建筑定义 ID 必须存在于 `DefinitionRegistry`；
5. 已处理命令 ID 不能重复；
6. 玩家命令序列必须单调递增；
7. 存档版本不能为空；
8. 模组列表在同一存档中必须可追溯。

## 七、多模型分工规则

### 7.1 核心框架设计者负责

以下内容只能由核心框架设计者修改或审核：

- `GameState`
- `Simulation`
- `CommandBus`
- `EventStream`
- `SaveSystem`
- `StableId`
- `DefinitionRegistry`
- `ModRegistry`
- `IGameSession`
- 联机协议
- 存档格式和迁移
- Tick 顺序
- 状态诊断工具
- 回放测试框架

### 7.2 DeepSeek 可负责

DeepSeek 适合执行：

- 按 schema 填建筑定义；
- 按模板补资源、配方、任务、事件数据；
- 批量整理 Markdown；
- 生成普通数据表；
- 补简单测试样例；
- 根据既定规则扩写建筑/NPC/事件说明；
- 做术语、格式、索引类低风险同步。

### 7.3 DeepSeek 禁止范围

DeepSeek 不得直接修改：

- 核心 Runtime 架构文件；
- 存档结构；
- 网络协议；
- ID 规则；
- 模组加载安全策略；
- 服务器权威校验；
- Tick 调度顺序；
- A级规则裁决。

若 DeepSeek 产物需要进入核心代码，必须先由核心框架设计者审查。

## 八、完成判定

核心框架 v1 完成必须满足：

- UI 可通过 `IGameSession` 驱动单机 Simulation；
- 所有玩家操作都能走 `CommandEnvelope`；
- 每个接受的 Command 都能产生 Event 或明确无事件原因；
- 存档可序列化、反序列化并修复缺失字段；
- 状态可计算 `StateHash`；
- 至少有一组回放测试能验证确定性；
- 模组 manifest 可被解析和校验；
- DeepSeek 分工边界已写入文档并被执行。

框架 v1.5 完成必须满足：

- Definition 数据有启动校验；
- M0 数据模组可加载；
- InvariantCheck 可在 Tick 后运行；
- 调试面板或日志能显示 Command -> Event -> StateHash；
- 核心系统有最小测试集。

框架 v2 完成必须满足：

- RemoteGameSession 可连接服务器；
- 服务器权威执行 Command；
- 客户端通过 Event/Snapshot 同步；
- 模组列表和 checksum 不一致时拒绝加入；
- StateHash 不一致时能请求快照修复；
- 断线重连不会破坏命令序列。

## 九、下一步执行顺序

1. 补 `IGameSession` 与 `LocalGameSession`；
2. 补 `ModRegistry` 与 manifest 数据结构；
3. 补 `StateDiagnostics`，包含 StateHash 和基础不变量检查；
4. 为现有建筑系统建立最小回放测试；
5. 再扩展资源系统、NPC系统和任务系统。
