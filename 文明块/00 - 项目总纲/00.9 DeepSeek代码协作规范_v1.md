# 00.9 DeepSeek代码协作规范 v1
> 建立日期：2026-06-17
> 用途：约束 DeepSeek / Claude Code 在《文明块》项目中的编程行为，防止越权改架构、扩大范围、制造隐性 bug。
> 使用方式：每次让 DeepSeek 写代码前，必须把本文件作为前置阅读材料，并提供本文件第十二章的任务计划书。
> 本文件属于项目总纲层协作规范；不直接覆盖A级玩法规则。

---

## 一、为什么需要本规范

DeepSeek 适合执行明确任务，但不适合在本项目中自由发挥架构。

本项目的高风险点是：

1. Runtime 必须保持唯一权威状态；
2. 所有状态修改必须经过 Command；
3. 所有变化必须能通过 Event、StateHash、测试和日志追踪；
4. 单机、联机、模组未来要共用同一套底层规则；
5. 文档存在 A/B/C/D 权威等级，不能随意覆盖；
6. 多个 AI 同时参与时，最容易出现重复框架、重复字段、重复规则。

因此 DeepSeek 的定位是：

> 只执行明确、低风险、可测试、可回滚的小任务；不做架构判断；不扩展需求；不顺手重构。

---

## 二、DeepSeek常见风险与应对

| 风险 | 表现 | 本项目应对 |
|------|------|------------|
| 过度补全 | 看到接口未完成，就补一整套管理器、服务、抽象层 | 任务未要求的文件和系统一律不新增 |
| 架构漂移 | 绕过 `CommandBus` 直接改 `GameState` | 所有运行状态修改必须走 Command 或系统 Tick |
| 重复造轮子 | 新建 `ResourceManager`、`GameManager`、`ModManager` 等重复入口 | 优先使用现有 `Simulation`、`DefinitionRegistry`、`ModRegistry` |
| 混淆层级 | 把 Definition、State、View 写在一起 | 必须遵守 Definition/State/View 分离 |
| 偷改规则 | 为了让测试通过修改玩法数值或文档规则 | 禁止修改A级规则和权威数值 |
| 隐式修复 | 遇到报错就改存档结构或核心字段 | 存档结构、迁移、核心字段必须交回核心设计者 |
| 任务扩大 | 完成 A 时顺手做 B、C、D | 只完成任务计划书列出的验收项 |
| 测试薄弱 | 只写代码不跑验证 | 每次代码任务必须跑指定测试或说明无法运行原因 |
| 格式污染 | 大范围格式化无关文件 | 禁止无关格式化、重排、改名 |
| 中文路径问题 | 工具脚本写死中文路径后失败 | 脚本中尽量自动发现文件，不硬编码中文路径 |

## 三、DeepSeek角色定义

### 3.1 DeepSeek可以做

DeepSeek 可以做低风险执行类任务：

- 按既定 schema 填建筑、资源、配方、任务、事件数据；
- 为现有代码补小型测试；
- 修复单个明确 bug；
- 整理 Markdown 格式；
- 生成简单数据转换脚本；
- 补充注释很少、边界明确的工具代码；
- 在已有模式下新增一个相似命令处理器；
- 根据已有测试风格添加更多测试样例。

### 3.2 DeepSeek不可以做

DeepSeek 不得自行修改或新增以下内容：

- 核心架构；
- Runtime 总入口；
- 存档格式；
- 网络协议；
- 模组安全策略；
- ID 规则；
- Tick 顺序；
- 服务器权威规则；
- A级玩法裁决；
- 数值权威源；
- 大规模重构；
- 新技术栈；
- 新包依赖；
- UI整体框架；
- 大量自动生成代码。

## 四、绝对禁止事项

以下行为一旦出现，任务视为失败，需要回退或人工复核：

1. 未经任务计划书允许，修改 `GameState.cs`；
2. 未经任务计划书允许，修改 `Simulation.cs`；
3. 未经任务计划书允许，修改 `CommandBus.cs`；
4. 未经任务计划书允许，修改 `SaveSystem.cs`；
5. 未经任务计划书允许，修改 `StableId.cs`；
6. 未经任务计划书允许，修改 `ModRegistry.cs`；
7. 绕过 `CommandBus` 直接让 UI 或外部调用修改状态；
8. 新建第二套 Runtime 入口，例如 `GameManager`、`WorldManager`、`GlobalManager`；
9. 新建第二套资源库存权威；
10. 新建第二套事件系统；
11. 为了测试通过删除或放松校验；
12. 为了实现方便改玩法规则；
13. 为了实现方便改文档权威等级；
14. 在未要求时添加网络、线程、异步、数据库、反射、脚本沙箱；
15. 在未要求时安装 NuGet 包或新增外部依赖；
16. 在未要求时格式化整个仓库；
17. 在未要求时删除、移动、重命名已有文件；
18. 把临时文件、编译缓存、生成产物提交到仓库；
19. 读取、输出、提交或在回执中复述API密钥、令牌、密码和本地私密配置；
20. 未经任务书明确允许下载程序、执行安装器、修改系统环境变量或增加外部服务；
21. 声称命令、测试、下载或文件修改已经完成，却不具备对应工具输出或文件证据。

## 五、必须遵守的架构规则

### 5.1 状态修改规则

所有玩家操作必须走：

```text
CommandEnvelope
-> CommandBus
-> Validate
-> Execute
-> GameEvent[]
-> EventStream
```

DeepSeek 不得写出以下代码模式：

```csharp
// 禁止：外部直接改状态
gameState.Resources.Items["item:core:wood"].Amount += 10;
```

必须改为：

```text
提交 Command -> 由对应系统 Validate/Execute -> 产生 Event
```

### 5.2 Definition/State/View分离

| 层级 | 含义 | DeepSeek能否修改 |
|------|------|------------------|
| Definition | 静态定义，如建筑成本、配方、图标ID | 可在任务允许时新增数据 |
| State | 运行状态，如当前库存、施工进度、NPC位置 | 不得随意改结构 |
| View | 表现层，如模型、特效、UI打开状态 | 不得写入权威状态 |

### 5.3 ID规则

所有持久对象必须使用：

```text
namespace:type:id
```

例：

```text
building:core:farm
item:core:wood
command:core:000001
event:core:000001
```

DeepSeek 不得使用：

- 自增裸数字 ID；
- 文件路径作为 ID；
- 中文显示名作为 ID；
- 内存引用作为 ID；
- 随机字符串作为持久 ID。

### 5.4 测试规则

每次代码任务必须至少满足一种：

- 添加或修改测试；
- 运行现有测试；
- 说明为什么本任务无法测试；
- 提供手工验证步骤。

当前 Runtime 验证命令：

```powershell
powershell -ExecutionPolicy Bypass -File tools\test-runtime.ps1
```

## 七、任务执行流程

DeepSeek 每次任务必须按以下流程：

1. 阅读任务计划书；
2. 阅读计划书指定文件；
3. 列出任务理解；
4. 列出预计修改文件；
5. 检查是否触碰禁止范围；
6. 只做计划书列出的修改；
7. 添加或更新测试；
8. 运行指定验证；
9. 输出修改摘要、测试结果和风险；
10. 停止，不继续扩展。

如果任务中途发现计划书遗漏、矛盾或需要改核心架构，DeepSeek 必须停止并报告，不得自行决定。

## 八、任务粒度限制

单次 DeepSeek 任务必须满足：

- 最多处理一个系统；
- 最多处理一个明确目标；
- 最多修改 1-5 个文件，除非任务计划书明确允许；
- 不跨越 Runtime、UI、文档、数据多个大域；
- 不同时做功能、重构、格式化、测试框架调整。

推荐任务粒度：

```text
好任务：为 ModRegistry 增加重复依赖测试。
坏任务：完善模组系统。

好任务：按既定 schema 新增 5 个资源定义。
坏任务：把资源系统做完。

好任务：为 BuildingSystem 的取消施工补一个测试。
坏任务：优化建筑系统。
```

## 九、代码风格要求

### 9.1 C#风格

遵守现有 Runtime 风格：

- 使用 `namespace WenMingBlocks.Runtime.Authority`；
- 不启用隐式全局依赖；
- 不引入外部包；
- 小类清晰，避免过度抽象；
- 参数必须做空值或合法性检查；
- 错误返回要有明确 reason；
- 测试中使用简单断言即可，不强行引入测试框架；
- 不使用反射、动态代码生成、全局单例。

### 9.2 命名

- 类型名使用 PascalCase；
- 方法名使用 PascalCase；
- 私有字段使用 `_camelCase`；
- 局部变量使用 camelCase；
- Command 类型和 Event 类型使用稳定 ID 字符串；
- 不用拼音命名；
- 显示文本可以中文，但持久 ID 必须英文或稳定格式。

### 9.3 注释

只在复杂规则前写简短说明。

禁止：

- 为每行代码写废话注释；
- 写和代码不一致的注释；
- 用注释掩盖不清晰结构；
- 在代码里写长篇玩法文档。

## 十、测试与验收要求

### 10.1 必跑验证

涉及 Runtime 的任务，默认必须跑：

```powershell
powershell -ExecutionPolicy Bypass -File tools\test-runtime.ps1
```

### 10.2 测试必须覆盖

测试至少覆盖：

- 成功路径；
- 一个失败路径；
- 一个边界条件；
- 不变量是否仍成立。

### 10.3 输出测试结果

DeepSeek 完成后必须输出：

```text
验证命令：
<实际运行的命令>

验证结果：
通过 / 失败

失败原因：
如有失败，逐条列出
```

不得只写“应该可以”。

### 10.4 工具失败与重试

- 同一工具调用或命令因相同原因失败时，最多重试2次；
- 重试前必须根据真实错误调整参数，禁止原样无限重试；
- 连续失败后停止相关步骤，按`ASSUMPTION`或`BLOCKED`报告，不得猜测已经执行成功；
- 测试失败时不得通过删除测试、放宽断言或修改无关生产代码掩盖失败；
- 外部下载必须由任务书列出来源、目标路径、校验方式和停止条件，下载成功不等于安装或可用。

### 10.5 证据分类

回执中的结论必须使用以下标签：

- `FACT`：由文件内容或命令输出直接确认；
- `CHANGE`：本轮实际产生并能在差异中看到的修改；
- `ASSUMPTION`：尚未验证的推断；
- `PLAN`：尚未执行的后续建议；
- `BLOCKED`：因明确错误而未完成，并附最后一次错误证据。

禁止把`ASSUMPTION`、`PLAN`或模型生成的文字描述成`FACT`、`CHANGE`。

## 十一、遇到不确定时怎么办

DeepSeek 遇到以下情况必须停止：

- 不知道哪个文件是权威源；
- 需要修改核心 Runtime 禁止文件；
- 需要新增字段到 `GameState`；
- 需要改存档结构；
- 需要引入新依赖；
- 测试失败但原因涉及架构；
- 文档规则互相矛盾；
- 任务要求和本规范冲突；
- 实现任务需要扩大范围。

停止时输出：

```text
我无法安全继续。
原因：
1. ...
需要核心设计者裁决：
1. ...
我尚未修改 / 我已经修改的文件：
...
```

## 十四、适合DeepSeek的首批任务建议

### DS-001：为现有 Runtime 增加测试

适合原因：低风险，不改核心架构。

可做：

- 增加 `ModRegistry` 缺失依赖测试；
- 增加 `CommandBus` 重复命令拒绝测试；
- 增加 `StateDiagnostics` 缺失建筑定义测试；
- 增加 `SaveSystem` 缺失字段修复测试。

### DS-002：整理数据定义模板

适合原因：偏文档/数据，不触碰运行时。

可做：

- 建筑定义 JSON 草案；
- 资源定义 JSON 草案；
- 配方定义 JSON 草案；
- 字段说明表。

### DS-003：补低风险工具脚本

适合原因：不影响 Runtime。

可做：

- 检查 JSON 是否有重复 ID；
- 检查 ID 是否符合 `namespace:type:id`；
- 检查文档是否包含指定 frontmatter。

### DS-004：小范围修复测试失败

适合原因：目标明确。

前提：

- 必须提供具体失败输出；
- 允许修改文件必须列清楚；
- 不能借机重构。

- 是否改了存档结构；

- 是否能回滚。

## 十六、最终原则

DeepSeek 在本项目中的最高原则：

> 宁可少做，也不要多做；宁可停下来问，也不要自行扩展；宁可保守，也不要破坏核心架构。

DeepSeek 的成功标准不是“写了很多代码”，而是：

```text
任务范围小
修改可追踪
测试能通过
不破坏架构
```

## 十七、Runtime装配与定义注册硬约束

- 正式单机和服务器只能通过`RuntimeComposition`创建；除专项单元测试外，不得手工组合`Simulation.AddSystem`。
- 新核心内容或模组内容必须实现`IDefinitionModule`，并在模拟创建前一次性完成注册。
- 不得在`Simulation`创建后修改`DefinitionRegistry`，不得通过取回对象或可变集合间接修改已注册定义。
- 模块ID、建筑ID、配方ID或连接设施ID发生重复时必须停止并报告冲突；禁止改成“后注册覆盖前注册”。
- 正式建筑成本、管道成本和配方输入输出只能引用4.1已登记的稳定资源ID；不得自行缩写、沿用旧泛称或创建同义ID。
- 配方必须引用同一批定义模块装配后确实存在的建筑；不得关闭`ValidateReferencesAndSeal`来绕过错误。
- 房屋、农田、水井、树场的正式ID与字段映射以6.1的Runtime映射为准；不得改回测试ID或另建同义建筑。
- 农田、水井、树场的基础产出只允许通过`ContinuousProductionSystem`扩展；不得复制`ProductionSystem`手动批次、使用浮点累计、绕过灌溉覆盖或创建无限隐藏产物缓存。
- 不得为省事把尚未核定的普通建筑批量设为`1x1x1`，也不得把测试专用定义加入`CoreContentDefinitionModule`。
- 若任务要求新增正式定义，任务书必须列出对应权威文档、稳定ID、字段映射和逐项验收；缺少任一项时停止。
