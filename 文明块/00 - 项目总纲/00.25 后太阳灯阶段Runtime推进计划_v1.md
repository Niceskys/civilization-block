## 二、总体推进原则

1. 先固化当前Runtime闭环，再扩展新玩法。
2. 先补可观察性和文档边界，再做系统性状态扩展。
3. 每次只推进一个系统，不同时跨Runtime、UI、文档和数值大域。
4. DeepSeek优先承担单文件测试、文档任务书和机械性核查。
5. Codex负责跨文件审查、架构边界判断、任务切分和提交前总验证。
6. 凡涉及`GameState.cs`、`SaveSystem.cs`、协议版本、Tick顺序、服务器权威边界，默认由Codex先审查，DeepSeek不得自行实施。

#### CX-NEXT-03 初步裁决准备记录（2026-06-28）

审查结论：暂不建议直接把树场接入Runtime光照裁决，应先交回核心设计者裁决“树场基础木材产出是否需要光照”。

已确认事实：

- `树场.md`只明确基础木材产出、应急炭化和燃料链边界，未单独定义树场光照停产状态。
- `太阳灯.md`的玩法摘要提到覆盖范围内的农田/树场在夜间或被遮挡时仍可产出，但当前Runtime接入状态明确只裁决农田。
- 当前`ContinuousProductionSystem`只对`building:core:farm`执行`HasRequiredAgriculturalLight`门控。
- 当前测试已有“Agricultural light only affects farm production”，确认树场被遮挡时仍保持生产；这属于现有Runtime边界，不应由后续执行模型顺手改掉。

利弊表：

| 方向 | 好处 | 风险 |
|------|------|------|
| 暂不接入树场光照 | 保持当前Runtime测试边界稳定；避免扩大玩法规则；不触碰木材与燃料恢复链 | 太阳灯摘要中的“树场”仍是后续待裁决内容，需要在任务书中反复标明未接入 |
| 接入树场光照 | 与太阳灯摘要中的“农田/树场”表述更一致；未来可形成农业/林业统一光照规则 | 会改变当前树场产出行为；可能影响早期木材供给、应急炭化和燃料恢复链；需要新增或改写测试 |

状态建议：

- 若核心设计者决定树场也受缺光影响，优先复用`paused_no_light`，因为它表达的是持续生产缺少必要光照，而不是农田专属状态。
- 不建议为树场新增状态，除非后续UI或诊断需要区分“农田缺光”和“树场缺光”的展示文案。
- 不要求太阳灯燃料必须先行；但如果担心树场被太阳灯无成本覆盖后影响木材/燃料平衡，应先裁决太阳灯燃料消耗，再接入树场。

后续任务边界：

- DeepSeek/Mimo可做只读核查：列出树场、太阳灯、生产链路中所有涉及“树场光照/燃料恢复链”的表述，不修改Runtime。
- Codex在核心设计者裁决后，再拆分是否修改`ContinuousProductionSystem`、测试矩阵和文档回写。
- 未经裁决，不新增树场光照Runtime实现，不改数值表，不改存档或协议。

#### CX-NEXT-04 初步裁决准备记录（2026-06-28）

审查结论：太阳灯燃料消耗不应作为小补丁直接实现，应先作为Runtime状态扩展任务单独裁决。原因是它不只是扣除`1燃料/日`，还需要定义太阳灯何时被视为“启用”、燃料覆盖如何跨Tick保存、缺燃料时覆盖是否失效，以及远端如何读取同一权威结果。

已确认事实：

- 该记录形成时，`太阳灯.md`快速摘要写有“运行消耗：1燃料/日”，但Runtime接入状态明确太阳灯燃料消耗仍未接入；后续已由`CX-NEXT-08`单独接入农田光照裁决所需的太阳灯燃料覆盖状态。
- `4.2 生产链路总览.md`已把垃圾处理场模式D与树场应急炭化产出的同一种“燃料”列为供应熔炉房/太阳灯的来源。
- 当前Runtime已有核心资源ID`resource:core:fuel`，也已有`building:core:sunlamp`。
- 当前太阳灯是否有效只看建筑定义、未损毁、耐久大于0和结构状态正常，不检查燃料。

建议裁决：

| 问题 | 建议方向 | 备注 |
|------|----------|------|
| 消耗资源 | 使用`resource:core:fuel` | 已存在核心资源ID，且燃料链文档明确太阳灯使用燃料 |
| 消耗数值 | 保持`1燃料/游戏日` | 这是建筑详情已有摘要，不在Runtime任务中改数值 |
| 消耗粒度 | 优先采用“预付1燃料获得1游戏日有效覆盖”的覆盖时长模型 | 类似农田灌溉覆盖，便于Tick切分一致和存档往返 |
| 缺燃料行为 | 缺燃料时太阳灯不提供光照覆盖 | 农田仍按缺光进入`paused_no_light`，不应另造农田状态 |
| 局部库存 | 需要先裁决是否允许太阳灯局部库存供燃 | 若允许，必须定义补给、锁定、拆除和摧毁行为 |
| 事件与诊断 | 建议新增可观察事件或诊断，而非只静默失效 | 便于后续UI提示读取Runtime结果 |
| 存档/协议 | 很可能需要升级 | 若新增每盏灯燃料覆盖剩余Tick或启用状态，必须进入存档、StateHash和联机快照 |

实施前必须回答：

1. 太阳灯是只在覆盖对象缺光时消耗燃料，还是只要建筑可运行就按日消耗？
2. 多盏太阳灯重叠覆盖同一农田时，是所有启用灯都消耗，还是只消耗完成覆盖所需的最小集合？
3. 缺燃料太阳灯是否仍参与遮挡、诊断和UI显示？
4. 燃料从全局库存扣除、太阳灯本地库存扣除，还是支持两者按固定顺序扣除？
5. 读档后燃料覆盖剩余时间如何恢复，是否允许恢复后立即补扣燃料？

后续任务边界：

- Codex先完成裁决与测试矩阵，再决定是否进入Runtime实现。
- Mimo可在任务书明确后承担中等范围实现，但不得自行修改存档/协议边界。
- DeepSeek更适合做只读核查、测试清单或单文件测试草案，不适合直接实现太阳灯燃料消耗。
- 在裁决完成前，不修改`GameState.cs`、`SaveSystem.cs`、`SimulationProtocol.cs`或`RuntimeComposition.cs`。

#### CX-NEXT-05 初步前置审查记录（2026-06-28）

审查结论：昼夜系统不应从UI或表现层先行实现；Runtime已经有权威`SimulationTick`，下一步应先把“由Tick派生昼夜阶段”的只读规则固化，再决定是否需要新增状态。当前不建议直接新增可变昼夜状态。

已确认事实：

- `1.3 时间系统规则.md`定义1游戏日为20现实分钟，白天/夜晚各占1/2，即白天10分钟、夜晚10分钟。
- Runtime当前定义`1 Tick = 1游戏秒`，`1游戏日 = 86,400 Tick`，1倍速每现实秒推进72 Tick，2倍速推进144 Tick，暂停推进0 Tick。
- `Simulation.Tick`只推进`GameState.SimulationTick`，并按已注册系统顺序执行；当前没有单独的昼夜阶段字段。
- 本记录形成时，农田Runtime光照只裁决遮挡与太阳灯恢复，不裁决夜晚；后续`a1e8565`已接入农田夜晚光照门控，完整昼夜行为系统仍未接入。
- `SimulationTick`已进入存档和StateHash；远端同步已有权威状态哈希路径。

最小状态草案：

| 项目 | 建议 | 原因 |
|------|------|------|
| 昼夜阶段来源 | 由`SimulationTick % GameTime.TicksPerGameDay`派生 | 避免保存第二套可漂移状态 |
| 白天区间 | 当日Tick处于前半日 | 对应1.3规则“白天10分钟” |
| 夜晚区间 | 当日Tick处于后半日 | 对应1.3规则“夜晚10分钟” |
| 新状态字段 | 暂不新增 | 当前固定昼夜比例可由Tick确定 |
| 新事件 | 可选：仅在跨越昼夜边界时发事件 | 若后续UI、怪物或NPC需要边界通知，再纳入任务 |
| 存档迁移 | 固定比例阶段不需要 | `SimulationTick`已保存，阶段可读档后重算 |
| 协议升级 | 仅派生读取不需要；若新增事件或可变昼夜参数则需要评估 | 避免无状态变更时扩大协议 |

首批测试清单：

1. `SimulationTick=0`派生为白天开始。
2. 半日边界前一Tick仍为白天，半日边界Tick进入夜晚。
3. 一日结束边界Tick回到下一日白天。
4. 暂停换算产生0 Tick，不改变昼夜阶段。
5. 大步长Tick跨越昼夜边界时派生结果与直接设置目标Tick一致。
6. 读档后仅凭`SimulationTick`恢复相同昼夜阶段，不需要补算退出时间。
7. 农田夜晚停产任务必须单独授权，不得在本前置审查中实现。

后续任务边界：

- DeepSeek可做只读测试清单整理或单文件派生规则测试草案。
- Mimo可在任务书明确后实现纯函数式昼夜派生 helper 和测试，但不得接入农田、NPC、怪物或UI行为。
- Codex负责裁决是否新增事件、是否接入农田夜晚停产、是否触碰协议版本。
- 未经裁决，不修改Tick顺序，不新增昼夜状态字段，不接入怪物/NPC夜班/UI渲染。

#### CX-NEXT-07 初步切分方案（2026-06-28）

已确认事实：

- `Simulation.Tick`先把`GameState.SimulationTick`推进到目标Tick，再按系统装配顺序执行各系统。
- `ContinuousProductionSystem`接收整段`deltaTicks`，当前按整段统一裁决工人、光照、输入、容量和效率。
- `ContinuousProductionSystem`位于建筑、NPC生命周期、住房和岗位系统之后，位于NPC生存、批次生产、垃圾生成和物流之前。
- 昼夜阶段当前只由`DayNightCycle`根据`SimulationTick`派生，不保存状态。

建议算法：

1. 在`ContinuousProductionSystem.Tick`内部计算`startTick = State.SimulationTick - deltaTicks`和`endTick = State.SimulationTick`。
2. 对农田生产推进使用分段循环，段边界至少包含昼夜半日边界和`endTick`。
3. 每段使用该段起点或段内任一Tick派生昼夜阶段；白天按现有遮挡/太阳灯规则裁决，夜晚必须有有效太阳灯完整覆盖。
4. 非农田持续生产暂不分段，保持当前整段逻辑，避免扩大到树场、水井和采掘。
5. 每段都复用现有输入覆盖、容量、效率余数、待转移产物和`paused_no_light`状态，不新增存档字段。
6. 若某段因缺光停止，后续同一Tick调用中不补算被暂停时间；恢复光照后只从后续Tick继续。

最低测试矩阵：

- 白天未遮挡农田仍正常产出并消耗灌溉水。
- 夜晚未遮挡且无太阳灯农田进入`paused_no_light`，不产出、不消耗灌溉水、不推进进度。
- 夜晚太阳灯完整覆盖农田可产出。
- 一个大步长Tick从白天跨入夜晚时，只结算白天段产出，夜晚段暂停。
- 一个大步长Tick从夜晚跨入白天时，夜晚段不补算，白天段恢复推进。
- 暂停0 Tick不改变状态。
- 读档后根据`SimulationTick`继续派生昼夜，不新增迁移。

实施边界：

- 不新增昼夜状态字段、事件、协议版本或存档版本。
- 本任务不接入NPC作息、夜班、怪物、UI、光照渲染、太阳灯燃料或树场光照；后续太阳灯燃料已由`CX-NEXT-08`单独接入。

4. Mimo完成后，核心设计者回到Codex线程；
5. Codex检查`git status`、`git diff`和验证输出；
6. Codex决定是否需要补救、提交或拆成下一任务。

停止条件：

- Mimo修改了任务书未允许的文件；
- Mimo需要改存档、协议、Tick或服务器权威边界；
- Mimo测试失败且原因需要生产代码裁决；
- Mimo输出缺少真实命令和退出码；
- Mimo试图继续下一个任务。
