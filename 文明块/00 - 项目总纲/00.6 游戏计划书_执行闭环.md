## 一、计划书定位

《文明块》当前不是缺少玩法设想，而是需要一个稳定的执行闭环：

1. 先确认游戏首发体验目标；
2. 再把机制裁决转成可回写任务；
3. 每次回写前检查冲突、权威源和受控摘要；
4. 回写后做一致性验证；
5. 最后更新冲突清单、路线图和审计索引。

本文件负责回答三个问题：

| 问题 | 本文件的回答 |
|------|--------------|
| 下一步先做什么 | 先处理DG设计裁决门，再处理P2中后期平衡，最后清理P3文档治理 |
| 怎么防止越改越乱 | 每个任务必须经过"裁决-回写-验证-收口"四步 |
| 什么时候算完成 | A级源文件已同步、冲突清单状态已关闭、审计索引已更新、校验方程成立 |

## 三、首发体验目标

### 3.1 核心定位

《文明块》的首发定位应是：

> 一个以方块建造、NPC居民、资源生产、防御压力和城市成长为核心的轻量沙盒经营游戏。

首发版本不追求最大系统量，而追求三个明确体验：

1. 玩家前10分钟知道自己为什么建、下一步做什么；
2. 玩家前30分钟至少经历一次建造完成、NPC入住、生产启动、昼夜变化和真实产出；
3. 玩家中期不会因为循环锁、无目标、升级无感或管理疲劳而流失。

### 3.2 首发必须守住的6条底线

来自`机制裁决与首发范围报告`：

| # | 首发必须项 | 目的 | 主要文档 |
|---|------------|------|----------|
| 1 | 升级节点新增功能解锁 | 修复升级无感 | 2.1、建筑单页、6.1 |
| 2 | NPC个性增强 | 修复NPC工具化 | 3.1、3.7、3.8 |
| 3 | 移除或削弱同种建筑效率惩罚 | 修复反成长体验 | 2.1、2.2、6.1 |
| 4 | 替代获取途径 | 修复燃料/科技循环锁 | 4.2、9.2 |
| 5 | UI操作深度压缩 | 修复管理疲劳 | 5.3、5.8、5.6 |
| 6 | 三层目标引导 | 修复前期/中期无目标 | 1.4、10.1、10.2 |

## 五、单任务闭环流程

每个任务必须按以下流程推进。

### 5.1 入口：任务立项

立项时必须填写：

| 字段 | 要求 |
|------|------|
| 任务编号 | 如P2-BATCH-03、DG-PREFLIGHT-01 |
| 对应冲突 | 必须引用C编号，或明确说明为新增治理任务 |
| 涉及文档 | 列出所有A级/B级/C级文件 |
| 权威字段 | 明确哪些字段以哪个文件为唯一权威 |
| 禁止范围 | 写清楚本次不能顺手改什么 |
| 旧存档影响 | 明确是否需要迁移或默认值 |
| 验收标准 | 写成可检查条目 |

### 5.2 执行：四步闭环

| 步骤 | 动作 | 产物 |
|------|------|------|
| 1. 裁决 | 对冲突做设计选择，必要时形成B级裁决 | 裁决记录 |
| 2. 回写 | 把裁决写入对应A级源文件 | A级文档变更 |
| 3. 验证 | 检查数值、引用、术语、受控摘要一致性 | 验证记录 |
| 4. 收口 | 更新00.2、00.3、00.4，移动历史状态 | 治理闭环 |

### 5.3 出口：完成判定

一个任务只有同时满足以下条件，才允许标记完成：

- A级源文件已经写入最终规则；
- 冲突清单中对应C编号已从活动区移入历史区，或状态已明确更新；
- 路线图中的任务映射已同步；
- 审计索引中的文件状态、备注和数量已同步；
- 如涉及数值，6.1/6.2/6.3/6.4或对应权威源已完成一致性检查；
- 如涉及UI或流程，前30分钟状态门、任务弹窗、提示来源不产生重复主动提示；
- 统计方程仍成立。

## 六、冲突检查清单

### 6.1 回写前检查

| 检查项 | 通过条件 |
|--------|----------|
| 权威等级 | A/B/C/D来源明确，没有用C/D直接覆盖A |
| 字段权威 | 每个数值、状态、触发条件都有唯一权威源 |
| 受控摘要 | 非权威文档中的重复数值标明来源 |
| 同步文件 | 所有会被影响的摘要、UI、数值表都列入同步范围 |
| 旧规则残留 | 旧方案、旧术语、旧数值不会与新规则并存 |
| 旧存档影响 | 需要迁移的状态字段已列出默认值 |
| 实现影响 | 需要程序配置、存档字段或Runtime命令的项目已列明 |

### 6.2 回写后检查

| 检查项 | 通过条件 |
|--------|----------|
| 编号一致 | C编号、PATCH编号、批次编号无跳号和重复 |
| 状态一致 | 00.2、00.3、00.4统计一致 |
| 术语一致 | 同一概念不再混用两个名称 |
| 数值一致 | 基础数值、UI显示、商品解锁、建筑详情不互相冲突 |
| 流程一致 | 玩家路径不会同时收到多个主动提示 |
| 边界一致 | 暂停、夜晚、拆除、满仓、死亡、旧存档都有处理 |
| 目标一致 | 沙盒自由、推荐目标、任务奖励不互相否定 |

### 6.3 重点高风险冲突

| 风险 | 典型表现 | 检查方式 |
|------|----------|----------|
| 任务清单化 | 沙盒游戏变成追任务 | 检查10.1、10.2是否使用"推荐"而非强制 |
| FOMO | 错过活动就损失一年或奖励 | 检查1.6、8.3、10.1是否有补领/循环 |
| 反成长 | 建得越多越被惩罚 | 检查2.1、2.2、6.1 |
| 循环锁 | 缺一个资源导致无法恢复 | 检查4.2、9.2、2.4、8.2 |
| UI重复打扰 | 同一行为同时弹任务、教程、推荐 | 检查5.2、5.8、10.3 |
| NPC工具化 | 性格只影响效率，没有行为差异 | 检查3.1、3.7、3.8 |
| 数值多源 | 同一价格/产量在多处维护 | 检查6.1、9.2、建筑详情、4.2 |

## 七、首发闭环验收

### 7.1 前30分钟验收

以`前30分钟补丁执行清单V3`为基准，首发前必须通过：

| 验收项 | 标准 |
|--------|------|
| 初始状态 | 新存档进入时暂停，放置首座建筑后时间启动 |
| 建造节奏 | 房屋和基础生产建筑可并发施工 |
| 首次产出 | 推荐路线下第一次真实资源产出不晚于T+25:00 |
| 第一夜 | 第一夜无怪物，夜晚不是强制等待 |
| 生产结算 | 多产物、输入不足、满仓、暂停都有一致规则 |
| NPC住房 | 初始无住所不会立即恐惧爆炸，D3后恢复正式规则 |
| 主线提示 | 状态门期间不弹窗，但任务面板可见、奖励正常 |
| 提示去重 | 同一行为只有一个主动提示来源 |

### 7.2 中期验收

| 验收项 | 标准 |
|--------|------|
| 升级动机 | 关键建筑Lv2/Lv3/Lv4至少有部分非纯数值变化 |
| 专业化 | 玩家集中发展某类建筑不会被硬性惩罚到无收益 |
| 替代路径 | 燃料、科技点、关键工具至少有高成本备用路径 |
| 管理成本 | 30人口时常用操作不超过2层主要路径 |
| NPC生命感 | 性格至少影响偏好、事件或行为表现之一 |
| 目标延续 | 发展篇章后仍有里程碑建议，不强制追踪 |

### 7.3 长线验收

| 验收项 | 标准 |
|--------|------|
| 沙盒自由 | 没有单一路线被标为唯一正确玩法 |
| 传承边界 | 传承是可选长线奖励，不等于强制通关 |
| 成就奖励 | 当前档奖励和跨档奖励边界明确 |
| 叙事入口 | 隐藏剧情可选，不阻断核心城市发展 |
| 存档持续 | 结局或传承后是否继续游玩有明确规则 |

## 八、文档治理闭环

### 8.1 每次修改后的同步文件

只要修改A级源文件，至少检查：

| 文件 | 检查内容 |
|------|----------|
| 00.2 全库规则冲突清单 | 对应冲突状态、活动/历史统计 |
| 00.3 正式源文件回写路线图 | 任务映射、批次状态、下一任务 |
| 00.4 全库Markdown审计索引 | 文件状态、备注、总数 |
| 00.1 文档权威与版本规则 | 如新增B级文件或权威原则需同步 |

### 8.2 归档规则

| 类型 | 处理 |
|------|------|
| 已被新版本替代的方案 | 移入D级历史说明，不再作为依据 |
| 已裁决但未回写方案 | 保持B级，标记"待回写" |
| 已回写裁决 | 在B级文件中保留记录，A级文件成为实际权威 |
| 诊断建议 | 保持C级，不直接执行 |
| 存根文件 | 明确写入"存根/指向文件/不可作为规则源" |

### 8.3 新增机制准入门槛

新增机制必须同时满足：

1. 不破坏方块建造、NPC管理、生存防御三角结构；
2. 不制造不可逆FOMO；
3. 不让玩家为了任务而不是城市本身游玩；
4. 不新增已有系统能解决的问题；
5. 有明确A级归属文件；
6. 有实现层入口：状态字段、命令、事件或配置项；
7. 有验收标准和失败回滚方案。

## 九、实现层衔接

Runtime架构蓝图已经提出：

- GameState统一状态树；
- CommandBus唯一修改入口；
- EventStream唯一输出；
- Definition/State/View三层分离；
- SeededRNG确定性随机；
- 版本化SaveSystem。

后续文档回写时，应逐步把规则转成实现可读字段：

| 规则类型 | 实现映射 |
|----------|----------|
| 建筑数值 | BuildingDefinition |
| 建筑实例状态 | BuildingState |
| NPC属性和生命周期 | NPCDefinition + NPCState |
| 生产链和配方 | RecipeDefinition + ProductionState |
| 任务与里程碑 | QuestDefinition + QuestState |
| 随机事件 | EventDefinition + SeededRNG |
| UI提示 | ViewState，不进入权威状态 |
| 存档迁移 | SaveSystem migration |

任何新规则如果无法映射到Definition、State、Command、Event或View之一，就不能进入实现阶段。
