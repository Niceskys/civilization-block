# 系统架构改造蓝图
> 阶段二：系统级重构框架
> 基于问题诊断报告 → 提炼为6大系统模块的可执行改造方向
> 本文件不包含具体数值、不修改原文档、不涉及代码实现

---

# 1. 留存驱动系统（Retention System）

## A. 当前问题总结

- 游戏无预设胜利条件，玩家前10分钟缺乏目标导向
- 前30日无危机保护期过长，玩家养成"无危机"预期后突然被摧毁
- 正向爆点事件概率极低（2%/日），大部分玩家体验不到惊喜
- 每日结算简报每20分钟打断一次，适得其反
- 夜晚无事件时10分钟纯空转，无内容可操作
- 连续3日跳过结算后可选择关闭，关闭后失去成长感知
- 月报在玩家无突破时展示"无变化"，强化"原地踏步"感受
- 地皮解锁需半小时现实时间，纯时间墙
- 蓝图分享系统仅首次有奖励，无长期参与动力
- 遗产提取上限导致"积攒到5000和500没区别"

## B. 核心结构缺陷

- **缺乏"短周期奖励节奏"**：玩家每3-5分钟应获得一次小奖励、每10-15分钟一次中奖励、每1-2小时一次大奖励。当前设计只有"每20分钟结算一次"的单节奏。
- **问题不是"没有内容"而是"内容在错误的时间出现"**：正向事件太少、太随机，而负向事件（危机）太晚出现。
- **"空转时段"未被填充**：夜晚空闲期没有被设计成"内容的空白"而是"等待的空白"。

## C. 改造方向

- 将"第一次吸引"从第0分钟到第10分钟设计为渐进式解锁序列，每30秒一个新发现
- 将"再次回来的理由"从"怕亏"转为"期待明天的惊喜"
- 将"长期玩下去的动力"从"自己定目标"转为"系统不断提供新鲜挑战"
- 将"空转时间"从缺点转化为"可选择的深度玩法时间"

## D. 必须新增机制

### 机制1：渐进式惊喜序列（Progressive Surprise Cadence）
- **作用**：确保玩家每3-5分钟获得一次小正反馈，拉长单次游戏时长
- **不增加新内容**，而是重新排列现有内容出现的时机和频率

### 机制2：回归引力系统（Return Gravity）
- **作用**：让玩家下线时"知道明天有什么在等他"
- 基于当前城市状态自动生成"明天的可期待事件"

### 机制3：空转填充层（Downtime Filler）
- **作用**：将夜晚等空转时段改造为"可选的深度沉浸时间"
- 提供非核心但有趣的可选操作，而非强制玩家等待

# 2. 核心循环系统（Core Loop System）

## A. 当前问题总结

- 采集→建造→升级的循环在中期断裂：升级只有数值提升无新功能解锁
- 同种建筑第4座仅40%效率，强制多样化但早期种类不足
- 上下层无相邻加成+高层惩罚，玩家无堆叠动机
- 堆叠越高惩罚越重，反成长设计
- 拆除中间层30分钟未修复即全毁且不返还，惩罚过度
- 工具变成持续维护支出而非一次性投资升级
- 地下挖掘纯机械劳动：投入时间→判定→获得固定资源，无分支选择
- 不同地皮地下独立计算，重复挖掘10层是重复劳动
- 燃料循环锁导致生产链在早期即可能卡死
- 科技点唯一来源是研究站，前10小时无科技
- 贸易路线Lv5门槛极高，大部分玩家达不到
- 垃圾处理4种模式切换无冷却，选择无意义

## B. 核心结构缺陷

- **循环缺乏"变奏"**：采集→建造→升级→再采集的循环中，每一步的结果都是可预测的。没有"这个决定可能导致意外收获"的不确定性。
- **惩罚＞奖励**：堆叠、重复建造、高层作业——这些行为的惩罚远大于奖励，玩家自然选择"不做"。
- **循环锁太多**：燃料锁、工具锁、科技点锁——这些锁不是"挑战"而是"等待"。
- **"更多"不等于"更好"**：建筑数量增加、层数增加、人口增加，带来的管理成本和惩罚也在增加，净收益递减。

## C. 改造方向

- 从"线性增长"改为"阶梯式突破"：不是每座新建筑都加一点点，而是每隔几座建筑解锁一个全新能力
- 从"惩罚多样化"改为"奖励专业化"：鼓励玩家在某个方向深耕，而非强制多样化
- 从"全手动管理"改为"渐进式自动化"：中期允许玩家"设置策略，让系统自动执行"
- 从"循环锁"改为"解锁捷径"：单一来源被卡时，提供替代途径（成本更高但可行）

## D. 必须新增机制

### 机制4：突破性升级节点（Breakthrough Nodes）
- **作用**：每隔一定数量的建筑/等级，不是"数值+5%"而是"解锁一个全新功能"
- 打破"升级只有数值变化"的单调感

### 机制5：专业化加成路径（Specialization Paths）
- **作用**：让玩家在"多种类发展"和"单种类深耕"之间做出有意义的选择
- 替代当前"不做多样化就惩罚"的负向设计

### 机制6：自动化策略层（Automation Strategy Layer）
- **作用**：中后期将重复性管理（工作分配、资源调配）从手动转为策略设置
- 玩家从"操作者"升级为"策略制定者"

### 机制7：替代路径系统（Alternative Routes）
- **作用**：为核心资源（燃料、科技点、工具）提供第二条获取途径
- 消除循环锁，但替代途径成本高于主途径

# 3. 世界活性系统（World Simulation System）

## A. 当前问题总结

- 生物群系单向影响生产效率，玩家行为无法改变群系
- 特殊地皮纯随机预设，玩家无法"创造"特殊地皮
- 天气80%时间无影响（晴天55%+阴天25%），纯装饰
- 危机事件只影响单一子系统，无"全城动员"感
- 恐惧值只与死亡事件相关，与城市生活质量无关
- NPC属性只影响效率，不影响战斗/贸易/特殊事件
- 好感度系统在人口<10时几乎不发挥作用
- 生育与城市环境无关，只检查食物/水数量
- 社交事件效果局限于NPC属性，不影响城市运营
- NPC死亡后无永久纪念（除5秒生平面板）
- 外部消息纯氛围不影响机制，玩家3次后不再看
- 贸易路线被动运作，不与城市发展互动
- 隐藏剧情与世界隔离，NPC不讨论、城市无变化
- 叙事进度与BOSS击败绑定，与发展方向冲突

## B. 核心结构缺陷

- **世界是"静态"的**：玩家的行为不会在世界上留下可见的、可感知的变化。群系不会变、NPC不会主动改变行为模式、事件不会因为城市而不同。
- **NPC是"功能组件"而非"居民"**：NPC有7个属性和好感度，但这些只影响"工作效率"，不影响"他们是谁"。
- **反馈是"单向"的**：系统影响玩家（天气减产），但玩家不影响系统（无法改良环境）。
- **故事和玩法是"两条线"**：隐藏剧情是独立附加层，与"玩家正在建造的城市"无关。玩家做一件事但看到另一个故事。

## C. 改造方向

- 从"被动世界"改为"响应式世界"：世界对玩家的行为做出可见的、持续的变化
- 从"NPC是工具"改为"NPC是角色"：赋予NPC可感知的个性，让他们"活"在城市中
- 从"单向影响"改为"双向互动"：玩家改变环境，环境回应玩家
- 从"故事附加"改为"故事嵌入"：叙事和游戏行为在同一轨道上推进

## D. 必须新增机制

### 机制8：世界状态层（World State Layer）
- **作用**：创建一个"世界记住玩家做了什么"的系统
- 城市规模、人口、科技水平等状态改变世界外观、事件内容、NPC对话

### 机制9：NPC个性引擎（NPC Personality Engine）
- **作用**：让每个NPC有可见的个人目标、偏好、习惯
- 从"属性数字"升级为"有行为的角色"

### 机制10：环境响应系统（Environmental Response）
- **作用**：玩家对环境的改变（建造、清除、污染等）触发世界的反向反馈
- 改变目前"单向影响"的结构

### 机制11：叙事嵌入层（Narrative Embedding）
- **作用**：将隐藏剧情线索嵌入到日常游戏行为中
- 玩家在"正常玩游戏"的过程中自然推进叙事，而非刻意去"做剧情任务"

# 4. 反馈与爽感系统（Feedback System）

## A. 当前问题总结

- 勘探60%结果是无障碍物（无奖励），做了没意义
- 升级到Lv3解锁自动运转但无特殊视觉反馈，玩家可能不知道
- 工具升级从Lv1→Lv3外观无变化，感受不到升级
- 地下挖掘产出自动飞入仓库，缺少"挖到宝"的兴奋感
- 科技点累积但长时间无法使用，每日看数字增长无反馈
- 恐惧值上升无即时视觉警告，玩家看不到
- NPC生育无命名/性别的仪式感，只是背景事件
- 建筑停产时玩家无法加速恢复，只能"等着"
- 资源溢出自动销毁无补偿，"做了白做"
- 手动交互点（点击加速播种等）隐藏在3层操作之下
- 挑战进度只在面板中可见，主界面无显示
- 任务完成通知无具体奖励描述，不知道得了什么

## B. 核心结构缺陷

- **反馈层级不足**：当前只有"数字变化"这一种反馈，缺少视觉、听觉、情绪三个维度的配合。
- **反馈时机不对**：正向反馈（升级/建造完成）太短太弱，负向反馈（资源不足/恐惧上升）太隐蔽太晚。
- **"做了有结果"但"结果不可见"**：玩家操作后数值变了，但UI中没有让玩家"感觉到"变化。
- **高交互功能被深埋**：建筑内部的手动交互点设计得很好，但需要"双击→全屏→找交互点"三步才能触达。

## C. 改造方向

- 每种重要操作至少获得2种反馈（数值+视觉/数值+音效/视觉+情绪）
- 正向反馈即时、显著、有"庆祝感"；负向反馈提前预警、可逆转
- 高价值交互从深层操作层级提到浅层（最多2次点击可达）
- 从"数字变化"升级为"可感知变化"

## D. 必须新增机制

### 机制12：三层反馈架构（Triple-Layer Feedback）
- **作用**：每个操作同时触发数值+视觉+情绪三个维度的反馈
- 定义"小反馈"/"中反馈"/"大反馈"的触发事件和表现强度

### 机制13：预告预警系统（Pre-warning System）
- **作用**：负向事件（资源将耗尽、恐惧将升级、建筑将损坏）提前一定时间以视觉方式预警
- 玩家有"反应时间"而非被动承受

### 机制14：操作可达性重整（Accessibility Restructure）
- **作用**：将高价值手动交互（加速/微调）移到主界面或建筑详情面板第一层
- 所有可交互操作 ≤ 2次点击可达

### 机制15：庆祝时刻生成器（Celebration Generator）
- **作用**：特殊事件（第一次产出某资源、人口达到整数、建筑升级等）触发定制化庆祝反馈
- 打破"每次升级都一样"的单调感

# 5. 节奏与压力系统（Pacing System）

## A. 当前问题总结

- 白天/夜晚严格各10分钟，节奏过于固定无变化
- 前1-5日怪物1-3只太容易→第21日5-12只混编太突然，节奏断裂
- 暴风雨罕见（5%）结束后还要强制阴天1天，无"天晴释放"
- 天气切换在后台进行，玩家感知不到切换过程
- 夜晚无预警（除屏幕泛红微光），建造中突然被打断
- 月报在玩家无突破时显示"无变化"，强化停滞感
- 动态难度调节惩罚防御好的玩家（连续无损→下夜+20%）
- BOSS威胁值惩罚建造多的玩家（每座建筑+1威胁值）
- 30日无危机保护期过长，前10小时无高潮
- 主线任务9个仅覆盖前5日，之后节奏空白

## B. 核心结构缺陷

- **节奏缺乏"呼吸"**：好的节奏是"紧张→释放→平静→再紧张"的交替。当前设计是"长时间平缓→突然爆发→再长时间平缓"。
- **压力来源单一**：所有压力来自怪物和资源短缺，缺少"选择压力"（做了A就不能做B）和"信息压力"（不知道会发生什么）。
- **没有"预期管理"**：玩家无法预判接下来会发生什么，所以"不知该准备什么"。知道"第21天会变难"和"不知道会变难"是两种完全不同的体验。
- **动态调节方向错了**：动态难度应该帮助陷入困境的玩家（降低难度），而非惩罚表现好的玩家。

## C. 改造方向

- 从"固定节奏"改为"响应式节奏"：根据玩家当前状态（资源、人口、建筑数）动态调整事件密度
- 从"单一压力源"改为"多维度压力"：资源、时间、选择、信息四个维度的压力交替出现
- 加入"可预期的不确定性"：玩家知道"可能会有事发生"但不知道"什么事什么时候发生"
- 动态调节改为"弹性调节"：偏负向时给予支持，偏正向时给予挑战

## D. 必须新增机制

### 机制16：节奏波形引擎（Cadence Wave Engine）
- **作用**：根据游戏内状态动态生成"平静波谷"和"紧张波峰"的交替
- 不是固定时间触发，而是基于玩家行为响应

### 机制17：多维度压力分布（Multi-axis Pressure Distribution）
- **作用**：将压力分布在资源/时间/选择/信息四个维度上，避免任何单一维度过载
- 当一个维度压力高时，其他维度自动降低

### 机制18：预期信号系统（Anticipation Signaling）
- **作用**：在重大事件/节奏转折前给出可识别的信号（非直接警告）
- 资深玩家可解读信号提前准备，新手也能事后理解"原来那个信号是这个意思"

### 机制19：弹性难度调节（Elastic Difficulty）
- **作用**：当玩家表现持续低于阈值时提供隐藏增益，高于阈值时增加精英变体
- 当前"惩罚强者"改为"托底弱者+挑战强者"

# 6. 目标与驱动系统（Goal System）

## A. 当前问题总结

- 游戏无预设胜利条件，前10分钟无目标
- 9个主线任务完成后（约前5日）再无引导
- 7个里程碑完成后（约50日/16.7小时）再无任何核心目标
- 沙盒后期"没有目标=无聊"
- 成就解锁的奖励只能在新档使用，削弱当前存档的成就感
- 日常任务需手动接取，错过即损失（FOMO设计）
- 周常只是日常的"升级版"，无差异化内容
- 隐藏BOSS解锁条件（80日/26.7小时）绝大多数玩家看不到
- 结局需要100日（33.3小时），核心玩家也难以达到
- 任务系统和世界观叙事完全无关

## B. 核心结构缺陷

- **目标层缺失**：缺少"3分钟目标""10分钟目标""1小时目标""10小时目标"的多层目标体系。当前只有"5日主线+50日里程碑"两个阶段。
- **目标是"一次性"而非"循环的"**：主线做完就没了，里程碑完成后就没了。没有"持续生成目标"的机制。
- **目标是"被动接收"而非"主动选择"**：玩家只能接系统给的任务，不能自己设定目标并让系统跟踪。
- **"为下一局而玩"的动机不足**：成就解锁的"下个档奖励"对当前档没有影响，玩家等于"白做"。

## C. 改造方向

- 建立"秒/分/时/日/周/月"六层目标体系，每层有不同的奖励类型
- 核心目标是"循环生成"的：当玩家完成一个目标后，系统根据当前状态生成下一个目标
- 加入"玩家自设目标"的支持：玩家可以标记"我要在10天内攒够X资源"，系统跟踪并奖励
- "当前档"的成就应该有"当前档"的奖励（非仅跨档奖励）

## D. 必须新增机制

### 机制20：六层目标结构（Six-Layer Goal Architecture）
- **作用**：定义从"秒级操作目标"到"月级长期目标"的完整层级
- 每层有独立的生成规则、奖励类型、UI表现

| 层级 | 时间尺度 | 目标类型 | 奖励 |
|------|---------|---------|------|
| L1 操作 | 3-10秒 | "把这座建筑放在这里" | 即时视觉反馈 |
| L2 任务 | 3-10分钟 | "今天产出够食物" | 星辉/资源 |
| L3 挑战 | 20-60分钟 | "堆叠到5层" | 特殊道具/解锁 |
| L4 里程碑 | 2-5小时 | "人口达到30" | 永久增益/建筑 |
| L5 成就 | 10-20小时 | "击败全部BOSS" | 跨档解锁/称号 |
| L6 叙事 | 20-40小时 | "完成剧情线" | 结局/传承奖励 |

### 机制21：动态目标生成器（Dynamic Goal Generator）
- **作用**：基于玩家当前状态自动生成"下一个合适的目标"
- 玩家完成一个目标后，系统在后台计算并推送下一个，永不断档

### 机制22：自设目标系统（Player-set Goal System）
- **作用**：允许玩家标记自己的目标（"我要攒500星辉"），系统跟踪进度并提供完成奖励
- 解决沙盒"我自己不知道要干嘛"的问题

### 机制23：当前档即时奖励通道（Current-save Reward Pipeline）
- **作用**：成就解锁时，当前存档立刻获得可见增益（非仅跨档）
- 解决"我为下一局而玩"的动力缺失

# 附：六大系统的关联关系

```
                     ┌─────────────────────────┐
                     │    留存驱动系统          │
                     │  (拉新 + 留人 + 回归)     │
                     └──────────┬──────────────┘
                                │ 提供 "我想继续玩" 的理由
                                ▼
┌─────────────────────────────────────────────────────────┐
│                     目标与驱动系统                       │
│             (我知道"为什么玩"和"下一步做什么")             │
└──────────┬──────────────┬──────────────┬───────────────┘
           │              │              │
           ▼              ▼              ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│  核心循环系统  │ │  世界活性系统 │ │  反馈与爽感系统 │
│ (我在做什么)  │ │ (世界在动吗) │ │ (我感受到了吗)│
└──────┬───────┘ └──────┬───────┘ └──────┬───────┘
       │                │                │
       └────────────────┼────────────────┘
                        │
                        ▼
               ┌────────────────┐
               │  节奏与压力系统  │
               │ (快/慢/紧/松切换)│
               └────────────────┘
```

- **留存驱动系统** 是外层，决定了玩家"想不想打开游戏"
- **目标与驱动系统** 是入口，决定了玩家"打开后知道做什么"
- **核心循环** + **世界活性** + **反馈爽感** 是执行层，决定了玩家"做得爽不爽"
- **节奏与压力系统** 是底层调控器，决定了玩家"什么时候爽、什么时候紧张"

# 执行建议

### 各系统的改造优先级

| 系统 | 优先级 | 原因 |
|------|--------|------|
| 目标与驱动系统 | **第一优先** | 无目标是当前最大的留存问题 |
| 反馈与爽感系统 | **第二优先** | 反馈不足是"不好玩"的核心原因 |
| 节奏与压力系统 | **第三优先** | 节奏断裂导致玩家在错误时间获得错误体验 |
| 核心循环系统 | **第四优先** | 核心循环的问题是"无趣"而非"不可运行" |
| 世界活性系统 | **第五优先** | 世界活性是"锦上添花"而非"雪中送炭" |
| 留存驱动系统 | **持续优化** | 留存是结果，由前5个系统的综合表现决定 |

### 23个新机制的实施建议

- **P0级机制**（必须先实现才能上线）：机制20（六层目标）、机制12（三层反馈）、机制21（动态目标生成）
- **P1级机制**（核心体验必需）：机制4（突破节点）、机制6（自动化策略）、机制14（操作可达性）、机制16（节奏波形）、机制18（预期信号）
- **P2级机制**（差异化竞争力）：机制2（回归引力）、机制9（NPC个性）、机制10（环境响应）、机制11（叙事嵌入）
- **可迭代机制**（后续版本添加）：机制3（空转填充）、机制5（专业化路径）、机制7（替代路径）

### 改造与非改造的分界线

| 可以改 | 不能改 |
|--------|--------|
| 事件触发时机和频率 | 删除/替换核心系统 |
| UI交互层级（路径深度） | 改变3D/体素的美术风格 |
| 反馈的维度和强度 | 改变"沙盒建造"的玩法类型 |
| 目标生成逻辑 | 改变"方块建造"的基础规则 |
| 节奏的动态调节 | 改变"NPC管理"的核心交互 |
| 系统的联动方式 | 改变"生存+建造+防御"的三角结构 |
