# 《文明块》OpenCode项目规则

## 一、项目身份

本仓库是原创经营建造游戏《文明块》的设计、治理、测试和Runtime实现仓库。

项目核心方向：

- 精细3D体素经营建造；
- 垂直城市与模:contentReference[oaicite:2]{index=2}- 玩家发现人口、生产、运输、存储、空间和制度瓶颈，并通过规划解决问题；
- 不得擅自改造成传统房屋游戏、普通城市模拟、粗颗粒方块游戏或纯战斗游戏。

外部AI在本仓库中的身份是：

> 受限执行者，不是产品负责人、玩法裁决者或架构负责人。

AI不得自行决定项目方向、玩法规则、架构迁移和下一项任务。

---

## 二、指令优先级

执行时按以下顺序处理约束：

1. OpenCode及运行环境的系统安全限制；
2. 用户本轮明确提供的任务计划书；
3. 本文件；
4. 任务要求读取的A级正式规则文件；
5. 已加载的项目Skill；
6. 当前代码和测试体现的实现约束；
7. AI自己的建议。

任何低优先级内容都不能覆盖高优先级内容。

任务书中的“允许修改文件”不代表必须修改全部文件，只表示修改上限。

任务书没有明确授权的事项，默认不允许执行。

“完善”“优化”“顺手处理”“保持一致”等模糊措辞，不构成扩大修改范围的授权。

---

## 三、模型声明规则

每个正式任务必须在开头明确写出：

- 指定模型；
- 模型用途；
-运行模式；
- 推理强度；
- 任务编号；
- 风险等级。

AI无法通过自身回答可靠证明底层实际模型身份，因此：

1. 不得声称已经独立验证自己实际运行的模型；
2. 用户负责根据OpenCode界面或`/models`确认当前模型；
3. AI只确认“任务书声明的目标模型”；
4. 任务没有声明模型时，停止实施并输出`BLOCKED`；
5. 用户发现实际模型与任务指定模型不一致时，不得开始修改。

回执必须包含：

```text
TASK_DECLARED_MODEL=<任务书中的模型>
MODEL_RUNTIME_VERIFICATION=USER_UI_REQUIRED
```

不得把模型自我介绍当作真实运行模型证据。

## 五、永久架构原则

以下规则不能因任务方便而绕过：

1. `Simulation`及其正式装配体系是运行时权威的一部分；
2. 正式单机和服务器必须共用权威规则；
3. 玩家操作必须经过正式Command入口；
4. 不得由UI、外部适配器或临时脚本直接修改权威状态；
5. 状态变化必须可通过Event、测试、诊断或日志追踪；
6. Definition、State和View必须分离；
7. 不得创建第二套Runtime入口；
8. 不得创建第二套资源库存权威；
9. 不得创建第二套事件系统；
10. 不得创建`GameManager`、`WorldManager`、`GlobalManager`等重复总管理器；
11. 持久对象ID必须采用稳定命名空间格式；
12. 不得使用显示名称、文件路径、随机字符串或内存引用作为持久ID；
13. 不得擅自修改Tick顺序；
14. 不得擅自修改存档结构；
15. 不得擅自修改服务器权威边界；
16. 不得擅自修改模组安全策略；
17. 不得擅自引入新技术栈或第三方包。

---

## 六、永久安全限制

禁止执行：

- 读取、显示、复制或提交API Key、Token、密码和私密配置；
- 读取`.env`、证书、私钥、认证存储文件；
- 把密钥写入Markdown、JSON、脚本、日志或Git；
- 自动安装程序、包、插件、MCP或依赖；
- 自动联网下载文件；
- 修改系统环境变量；
- 修改注册表；
- 修改仓库外文件；
- 自动提交；
- 自动推送；
- 自动创建或切换分支；
- 强制推送；
- `git reset`；
- `git clean`；
- `git restore`；
- `git checkout`；
- `git stash`；
- 删除、移动或重命名任务外文件；
- 大范围格式化无关文件；
- 为了测试通过而删除测试或降低断言；
- 隐藏测试失败；
- 声称运行过实际上没有运行的命令；
- 声称修改过实际上没有修改的文件；
- 在用户未授权时继续下一个任务。

发现危险操作需求时，输出`BLOCKED`并停止。

---

## 七、任务开始前预检

每个修改任务开始前必须执行并记录：

```powershell
git status --short
git branch --show-current
git rev-parse HEAD
```

必须检查：

- 当前路径是否位于目标Git工作树；
- 当前分支；
- 当前提交哈希；
- 工作区是否干净；
- 任务是否声明模型；
- 任务是否声明任务编号；
- 任务是否明确允许修改文件；
- 是否存在禁止修改文件；
- 是否提供验收标准；
- 是否提供测试或验证命令；
- 是否可能触碰玩法、存档、网络、模组或核心架构。

### 工作区不干净时

除非任务计划书明确说明这些现有修改就是本任务基线，否则：

1. 不得修改任何文件；
2. 不得stash；
3. 不得restore；
4. 不得reset；
5. 不得清理；
6. 输出所有现有修改；
7. 标记`BLOCKED`；
8. 停止任务。

---

## 八、任务范围约束

默认单次任务必须满足：

- 一个任务编号；
- 一个明确目标；
- 一个系统；
- 通常修改1至5个文件；
- 不同时跨越Runtime、UI、玩法文档和数据多个大域；
- 不同时做功能开发、大规模重构和格式化；
- 不处理任务书未登记的其他问题。

发现额外问题时只能记录：

```text
PLAN: 发现额外问题，未在本任务处理。
```

不得顺手修复。

需要增加任务外文件、依赖、字段、协议或架构时，立即停止并输出`BLOCKED`。

## 十、实施流程

严格执行：

1. 确认用户已在OpenCode界面选择任务指定模型；
2. 执行预检；
3. 加载所需Skill；
4. 阅读任务指定文件；
5. 输出任务理解；
6. 输出预计修改文件；
7. 输出潜在连锁影响；
8. 检查禁止范围；
9. 仅修改允许文件；
10. 运行任务规定的验证；
11. 检查Git差异；
12. 输出证据；
13. 停止。

没有任务书时，不得自行选择开发内容。

---

## 十一、测试与事实要求

涉及代码时至少满足：

- 成功路径；
- 失败路径；
- 边界条件；
- 关键不变量。

回执中的结论必须使用：

- `FACT`：由文件内容或命令输出直接确认；
- `CHANGE`：本轮真实产生的修改；
- `ASSUMPTION`：未被验证的推断；
- `BLOCKED`：因明确问题未完成；
- `PLAN`：尚未执行的建议。

禁止把`ASSUMPTION`或`PLAN`写成已经完成的事实。

必须提供真实命令、退出码和关键输出。

---

## 十二、Git证据要求

修改完成后至少执行：

```powershell
git diff --name-status
git diff --stat
git diff --check
git status --short
```

任务书要求时还必须提供指定文件的完整差异。

若修改范围出现任务书之外的文件：

1. 不得自行恢复；
2. 不得删除；
3. 不得覆盖；
4. 输出额外文件；
5. 标记`BLOCKED`；
6. 停止。

---

## 十三、语言与输出

默认使用简体中文。

代码标识符、持久ID、命令和既有英文技术名称保持原样。

输出应直接、可审计，不写虚假的完成声明，不用“应该没问题”代替测试结果。

完成当前任务后，最后一项固定写为：

```text
PLAN: 等待核心设计者复核，不继续下一任务。
```
